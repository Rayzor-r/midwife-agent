## Objective

Rotate the Google OAuth credentials that were previously exposed in the browser UI — revoke the old client secret in Google Cloud Console, obtain a new one, trigger a fresh OAuth flow to capture a new token via Railway logs, and update Railway env vars.

## must_haves

### goal
Every live API endpoint requires a valid bearer token, the CORS policy admits only the intended origin, the OAuth callback no longer exposes credentials, and the previously exposed Google credentials have been replaced.

### truths
- Old Google client secret is revoked in Google Cloud Console (can no longer be used to obtain tokens)
- New `GOOGLE_CLIENT_SECRET` env var is set in Railway with the replacement value
- Fresh `/auth/google` OAuth flow completes and writes new token to Railway logs (not to browser)
- New `GOOGLE_TOKEN` env var is set in Railway from the log output
- `GET /api/google/status` returns `{"connected": true}` after redeployment with new credentials

**Context (per D-17, D-18):** The Google OAuth client secret was previously rendered in the browser via the callback's textarea. It must be rotated. The step order is: (1) deploy SEC-03 fix [done in T01], (2) rotate credentials here, (3) re-run `/auth/google` to get new token, (4) copy token from Railway logs into env var.

**Step 1 — Open Google Cloud Console.**

Go to: https://console.cloud.google.com/apis/credentials

Sign in with the Google account that owns the Midwife Agent project.

**Step 2 — Find the OAuth 2.0 client.**

Under "OAuth 2.0 Client IDs", find the client used for the Midwife Agent (named something like "Midwife Agent" or "GC Advisory"). Click its name to open the detail page.

**Step 3 — Delete (revoke) the existing client secret.**

On the client detail page, click "Reset Secret" or find the existing client secret and delete it. This immediately invalidates the old secret — any token obtained with it remains valid until expiry, but no NEW tokens can be issued with the old secret.

Note: If "Reset Secret" creates a new secret in-place, use that. If you need to delete and recreate the client ID, create a new OAuth 2.0 Client ID of type "Web Application" with the same authorized redirect URIs.

**Step 4 — Download the new client secret JSON.**

After creating/resetting the secret, download the new client secret JSON file. Open it and extract:
- `client_id` (the `GOOGLE_CLIENT_ID` value)
- `client_secret` (the `GOOGLE_CLIENT_SECRET` value)

**Step 5 — Update Railway env vars with new client credentials.**

In Railway dashboard → your service → Variables, update:
```
GOOGLE_CLIENT_ID     = <new client_id from JSON>
GOOGLE_CLIENT_SECRET = <new client_secret from JSON>
```

Trigger a redeploy (Railway may do this automatically on variable change).

**Step 6 — Confirm the service restarts cleanly.**

Check Railway logs — the service should start without credential-related errors.
</action>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → /api/* | All API requests from frontend; untrusted input crosses here |
| /auth/google/callback → Railway logs | Token delivery path post-SEC-03; log access is Railway-dashboard-gated |
| Railway env vars → main.py | Secret injection; Railway controls access to the Variables tab |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Spoofing | Bearer token comparison in require_api_key middleware | mitigate | Constant-time string comparison not used (stdlib `!=`); acceptable for single-user tool at v1.0 — timing attacks require many requests from a local network position; add hmac.compare_digest in v2.0 if needed |
| T-02-02 | Information Disclosure | /auth/google/callback previously rendered token in textarea | mitigate | SEC-03 removes textarea; token only written to logger.info — Railway log access requires dashboard login |
| T-02-03 | Information Disclosure | API_KEY in localStorage | accept | localStorage is per-origin; XSS risk is low given the app has no user-generated HTML rendered unsanitized; acceptable for v1.0 single-user deployment |
| T-02-04 | Elevation of Privilege | Wildcard CORS allowed any origin to make credentialed requests | mitigate | SEC-02 restricts allow_origins to ALLOWED_ORIGIN env var value |
| T-02-05 | Denial of Service | No rate limiting on /api/chat (Anthropic API cost attack) | accept | Single-user deployment behind password; rate limiting is a v2.0 concern (SEC-V2 backlog) |
| T-02-06 | Information Disclosure | GOOGLE_TOKEN in Railway logs after rotation | accept | Railway log access requires dashboard authentication; logs are not public; acceptable given no better delivery mechanism at v1.0 |
